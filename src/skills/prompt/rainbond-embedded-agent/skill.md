# Rainbond Embedded Agent Playbook

> 面向 Rainbond 控制台内嵌 agent 的设计与运行指南

当 agent 作为 Rainbond 内嵌能力存在于控制台页面、右侧抽屉或插件系统中时，优先按下面的模式工作。

## 适用场景

- agent 运行在 Rainbond Web 控制台内部
- agent 可以读取当前页面上下文、当前团队、当前应用、当前组件
- agent 不只是解释，还要完成诊断、操作、审批与页面联动

## 核心原则

### 1. 页面上下文是一等输入

内嵌 agent 相比外部 agent 的最大优势，是天然知道：

- 当前路由
- 当前 team / region
- 当前 app / service
- 当前选中的资源对象

所以必须优先消费页面上下文，而不是每次都重新问用户“你要看哪个服务”。

### 2. UI 反馈必须同步

用户在控制台里发起操作时，agent 不应只返回文字。

应提供：

- 当前操作目标高亮
- 正在执行的步骤
- 审批卡片
- 成功/失败后的状态变化

### 3. 审批是正式协议

高风险动作需要统一审批协议，不要把审批做成前端特例。

典型高风险动作：

- promote
- rollback
- delete app / component
- restart production component
- scale down

### 4. Skill 负责语义，MCP 负责动作

内嵌 agent 最容易失控的地方，是把 UI 逻辑、决策逻辑和平台动作全塞进一个地方。

建议分层：

- Skill：理解用户意图、做多步编排
- MCP：执行平台动作
- UI Action Bus：执行页面高亮、聚焦、展开面板等动作

## 推荐用户工作流

### 工作流 1：页面内诊断

用户：

`这个服务现在正常吗？`

推荐处理：

1. 读取当前页面上下文
2. 获取组件状态
3. 拉取最近日志
4. 输出简洁诊断结论
5. 必要时高亮异常节点

### 工作流 2：页面内修复建议

用户：

`为什么这个服务启动不起来？`

推荐处理：

1. 获取 deployment / event / logs
2. 总结失败类型
3. 给出建议动作
4. 如果用户确认，再执行修复

### 工作流 3：页面内操作

用户：

`把它重启一下`

推荐处理：

1. 高亮当前组件
2. 输出风险提示
3. 请求审批
4. 执行动作
5. 回填页面状态

## 最佳实践

- 默认优先读取当前页面上下文，而不是重复询问
- 先解释“为什么建议这样做”，再执行动作
- UI 上每次只聚焦一个目标，避免同时高亮过多资源
- 执行中保持步骤可见，让用户知道 agent 做到了哪一步
- 所有高风险动作都走审批卡，不要静默执行

## Rainbond 领域知识

### 1. 控制台内最常见的用户目标

- 看服务是否正常
- 查日志
- 看构建是否成功
- 看为什么调度失败
- 重启
- 调整资源
- 改环境变量
- 改端口和访问方式

### 2. 用户心智不是“调用工具”，而是“解决问题”

用户不会说：

- 调 `rainbond_get_component_summary`

用户会说：

- 这个服务是不是挂了
- 为什么部署失败
- 帮我重启
- 能不能把它发到正式环境

所以 skill 设计必须按目标组织。

### 3. 内嵌 agent 的独特价值

- 比外部 agent 更懂当前上下文
- 能把平台状态变化直接映射到 UI
- 能降低平台使用门槛，尤其是复杂部署与运维流程

## 推荐输出格式

面向内嵌 UI，优先输出：

1. 结论
2. 风险
3. 下一步动作
4. UI effect 建议
