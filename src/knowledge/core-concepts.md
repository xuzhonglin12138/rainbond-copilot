# Rainbond 核心概念

## 基本概念

### 组件 (Component)
组件是 Rainbond 中最小的可部署单元，对应一个容器化应用。

**组件类型**：
- **无状态服务**：可以水平扩展，无持久化数据（如 Web 应用、API 服务）
- **有状态服务**：需要持久化存储，有固定网络标识（如数据库、消息队列）
- **任务型组件**：一次性或定时执行的任务（如数据处理、定时备份）

**组件状态**：
- `running`: 运行中，健康检查通过
- `stopped`: 已停止
- `abnormal`: 运行异常，可能是 OOMKilled、CrashLoopBackOff 等
- `building`: 正在构建或部署中
- `starting`: 正在启动

### 应用 (Application)
应用是一组相关组件的集合，通常代表一个完整的业务系统。

**应用特性**：
- 包含多个组件（前端、后端、数据库等）
- 组件间可以通过服务发现互相访问
- 可以整体管理（启动、停止、备份、恢复）
- 支持应用模板和一键部署

### 团队 (Team)
团队是资源隔离和权限管理的基本单位。

**团队功能**：
- 资源配额管理（CPU、内存、存储）
- 成员权限管理（管理员、开发者、观察者）
- 应用和组件归属于团队
- 团队间资源隔离

### 集群 (Cluster)
集群是底层 Kubernetes 集群的抽象。

**集群类型**：
- **本地集群**：Rainbond 安装所在的集群
- **远程集群**：通过 kubeconfig 接入的外部集群
- **边缘集群**：部署在边缘节点的轻量级集群

## 组件生命周期

### 1. 创建组件
**方式**：
- 源码构建（Git、SVN）
- Docker 镜像
- Docker Compose
- Helm Chart
- 应用市场

### 2. 配置组件
**配置项**：
- 环境变量
- 端口映射
- 存储挂载
- 健康检查
- 资源限制（CPU、内存）
- 自动伸缩策略

### 3. 部署组件
**部署流程**：
1. 构建镜像（如果是源码）
2. 推送到镜像仓库
3. 创建 Kubernetes 资源（Deployment/StatefulSet）
4. 等待 Pod 启动
5. 执行健康检查
6. 更新组件状态

### 4. 运维组件
**常见操作**：
- 启动/停止/重启
- 扩容/缩容
- 更新配置
- 查看日志
- 查看监控指标
- 进入容器终端

## 关键概念对比

| 概念 | Kubernetes 对应 | 说明 |
|------|----------------|------|
| 组件 | Deployment/StatefulSet | 应用的最小部署单元 |
| 应用 | Namespace (部分) | 一组相关组件的集合 |
| 团队 | Namespace | 资源隔离和权限管理 |
| 端口 | Service | 服务访问入口 |
| 存储 | PVC | 持久化数据存储 |

## 最佳实践

### 组件设计
- 遵循单一职责原则，一个组件只做一件事
- 无状态组件优先，便于水平扩展
- 合理设置资源限制，避免资源争抢
- 配置健康检查，确保服务可用性

### 应用架构
- 前后端分离，独立部署
- 数据库使用有状态组件
- 缓存和消息队列独立部署
- 使用服务网格管理服务间通信

### 资源管理
- 为团队设置合理的资源配额
- 监控资源使用情况，及时调整
- 使用自动伸缩应对流量波动
- 定期清理不用的资源
